## Verdict

The architecture is coherent and its central bet is validated by the references: the
discrete-targets contract genuinely removes the need for JSI, and the costs fx avoids are
real and measurable in Reanimated's own repo. The three biggest risks are not the
architecture itself:

1. **Product sequencing** — the demand center of this category (scroll/gesture-linked
   presentation) is deferred to V2 while V1 ships decorative effects.
2. **Two implementation idioms diverge from the Expo patterns fx is built on** — the
   hosted substrate rebuilds its hosting controller per prop batch, and every
   `FxSurfaceView` allocates Metal whether or not it draws. Each is invisible today and
   bites exactly when the next planned feature lands (the eased-uniform `transition`
   channel; motion on lists).
3. **An undocumented scope ceiling on presence** — exit animations cannot survive
   navigator pops under the JS-held model, and the `sheet`/`modal` preset names actively
   invite that exact use.

A fourth, found on the second pass: **the two platforms have already drifted** — Android's
hosted view fixed a defect iOS still has, and Android's shipped mechanics no longer match
the Compose vocabulary in the spine docs.

## What the references validate — keep these

- **The boundary ADR (`0-spine/05`) survives adversarial comparison.** Reanimated's price
  for its sync/worklet boundary, measured in its repo: ~232 C++ files / 23k+ LOC, a
  `compatibility.json` pinning each 4.x release to exact RN minors (4.5.x → RN 0.83–0.86)
  plus an exact worklets version, mandatory New Arch, and a Fabric commit hook
  (`ReanimatedCommitHook.h`) to fight prop clobbering. Nitro's 60× sync-call benchmark
  (7.27 ms vs 434 ms per 100k calls, its own docs) is irrelevant to a library that crosses
  the boundary once per state change. The `05` falsification test already passed.
- **The Fabric-invisible intermediate container** (`packages/ios/FxSurfaceView.swift:55-58`)
  is the cheap correct answer to the prop-clobbering problem Reanimated solves with C++
  commit hooks. Device-verified (U4-001/002).
- **The ownership spine (`04`) independently rediscovers the references' scar tissue.**
  RNGH injects `collapsable: false` via `cloneElement` so its wrapper survives view
  flattening (`Wrap.tsx:21`); Reanimated defers unmount via
  `setShouldAnimateExitingForTag` (`LayoutAnimationConfig.tsx:58-70`). fx designed both
  lessons in upfront.
- **Rive is the existence proof of fx's contract** — discrete inputs in
  (`setInputState`/`fireState`), native runtime owns frames, semantic state-change events
  out — shipping in production RN apps today.
- **The builder idiom is where the ecosystem ended up.** RNGH deprecated
  component-per-recognizer in favor of `Gesture.Tap().numberOfTaps(2)`; Reanimated ships
  `FadeIn.duration(300).springify()`. `fx.motion.*`/`fx.effect.*` starts where they ended.
- **Native value-equality on props (SURF-010)** removes a memoization footgun class
  Reanimated users carry.
- **The compatibility window is a marketing asset, not just an internal virtue**: fx's
  window is "whatever expo-modules-core supports," against Reanimated's exact-minor
  pinning. State it in the README at ship time.

## Architecture findings

### F1 — iOS hosted substrate rebuilds the hosting controller per prop batch (CRITICAL)

`OnViewDidUpdateProps` → `applyResolvedConfig()` → `mountHost()` tears down and recreates
the `UIHostingController` unconditionally (`packages/ios/FxModule.swift:32-34`,
`packages/ios/FxHostedView.swift:43-59, 79-95`). Three independent comparisons say this is
wrong:

- Expo's own `SwiftUIHostingView` keeps one persistent controller and pushes updates
  through an `@ObservedObject` props holder
  (`references/expo/.../SwiftUIHostingView.swift:45-101`).
- expo-glass-effect mutates a persistent `UIVisualEffectView` in place and animates the
  change (`references/expo/.../GlassView.swift:133-141`).
- **fx's own Android tier already ships the fix**: `FxHostedView.kt:66-75` keeps
  `mountedEffectId` and, when only `intensity` changes, mutates the live child — its
  comment names the exact failure mode ("would flash a blank frame, reset the shader
  clock, and reparse the AGSL each tick").

The iOS time-wrap (`FxShaderView.swift:38-39` derives time from the absolute date) papers
over the clock-reset symptom but not the structure: the planned native-eased uniform
`transition` channel (`3-motion/40` — "the DX heart of fx") cannot work on a substrate
that destroys the SwiftUI tree on every uniform change; symbol-effect state dies
mid-animation; glass press state resets. Severity CRITICAL because it blocks a ratified V1
channel, not because anything visibly fails today.

**Alternative:** persistent controller + observable props holder, per the Expo template
and the Android sibling. Mechanical; no architecture impact.

### F2 — eager Metal on every `FxSurfaceView` (HIGH)

`init` unconditionally runs `setUpMetal()` — command queue, `MTKView`, library resolution
(`packages/ios/FxSurfaceView.swift:61-66, 83-98`). Per DOC-014, presence/state/press
components all ride `FxSurfaceView`, so in V2 `<FxView preset="lift">` around 50 list
cards allocates 50 MTKViews to animate 50 transforms. RN's native driver allocates nothing
per view; Rive's per-view renderer is justified because the canvas is the content.

**Alternative:** allocate Metal lazily on the first non-empty `shader` prop (the
`pendingShader == ""` guard already exists for drawing — move it to allocation), or split
a Metal-free motion host from the shader surface. Related (F11): `device`/`commandQueue`/
`pipelineCache` are per-instance (`FxSurfaceView.swift:37-40`) — share them statically
before the list story.

### F3 — the manifest data does not exist; five dispatch renderings, zero lockstep (HIGH)

`packages/src/manifest/` holds the schema, `select()`, and tests — no `CapabilityManifest`
instance. Nothing at runtime consults a manifest; the native string switches are the only
dispatch (`FxHostedView.swift:63-75`, `FxShaderView.swift:66-80`, the Kotlin twins). The
catalog is therefore encoded in five unsynchronized places: schema-typed data (absent),
the `ShaderId` union (`effects/catalog.ts`), the Swift switches, the Kotlin switches, and
the `structure.{ios,android}.md` prose. `02` decision 1 says "generated-from or
lockstep-with"; nothing generates and nothing asserts. This is audit G1, still open —
compounded since the audit by `architecture.md` (per DOC-014) pointing at
`manifest/index.ts` as "the data" when that file is pure re-exports.

**Alternative:** author the manifest data as its own task; add a Jest conformance test
walking manifest ids ↔ `ShaderId` ↔ a checked-in (or generated) native id table. Codegen
(the Nitro/nitrogen model) only if node count grows.

### F4 — shipped Android mechanics contradict the spine's Compose vocabulary (HIGH)

The docs define `hosted` as the "SwiftUI/Compose host" (`research/README.md`,
`0-spine/01`) and `02`'s worked examples lower Android shaders via
`applyVia: 'ShaderBrush'` / Compose with `clock: 'frame-nanos'`. The shipped code
deliberately avoids Compose: plain `View` + `RuntimeShader` through `Paint.onDraw` + a
`Choreographer.FrameCallback` clock (`FxShaderView.kt:11-26`), with a sound argument for
why `RenderEffect` is wrong for generative shaders. The reasoning is good; the authority
chain is broken — `structure.android.md` may carry the recorded path (REAL-003), but the
spine docs above it still teach Compose, and the next agent will re-derive "Android hosted
= Compose" from them. Per the repo's own rule, a mechanic lives in exactly one place.

**Alternative:** a `rework` task reconciling `01`/`02` examples/README substrate wording
to the shipped realization (or explicitly recording View-based hosting as the Android
hosted mechanic with Compose as a future rung).

### F5 — presence has a scope ceiling the docs do not state (HIGH)

fx defers unmount by holding the exiting child in a ref and asking React to unmount later
(`04` seam 5, `35`) — the Framer/Moti JS-held model, which only works while fx's own
component stays mounted. Reanimated's native-held model
(`shouldAnimateExitingForTag_`, the deferred-animations queue, web-side DOM cloning)
exists precisely to cover what JS-holding cannot: exits when the whole subtree dies at
once — a navigator pop, a parent conditional, list eviction. fx's choice is right for its
constraints (no commit hooks, rule #7), but `preset="sheet"` and `preset="modal"` are
named after screen-scale presentations users will put directly under navigators, where the
handshake can never fire.

**Alternative:** no architecture change — document the boundary now ("presence animates
conditional rendering inside a mounted screen; it does not animate navigator
transitions") and re-examine the `sheet`/`modal` names against it. Audit `35` against
Reanimated's edge-case inventory before U7-001 (see F12).

### F7 — regime C sequencing (MEDIUM)

Reanimated's moat, verified in source, is exactly what fx forbids until V2:
`useAnimatedScrollHandler` and gesture worklets on the UI thread with zero JS-thread
crossings per frame. fx's `source` driver (native scroll read; render-server fidelity via
`scrollTransition` on iOS hosted) is architecturally better for its scope and Rive-like —
but it is V2, and "presentation runtime that cannot bind presentation to scroll" is how a
2026 RN developer will evaluate V1.

**Alternative:** pull the cheapest `source` rung forward — iOS-hosted
`scrollTransition`/`visualEffect` only, Android deferred. Additive, small, and it changes
the V1 story from "decorative" to "reactive."

## API design findings

### F6 — per-effect typed config is the front door and the least-typed part (HIGH)

`00` rules in "an agnostic effect catalog with deep typed props"; `02` carries
`UniformSpec` for exactly this; expo-mesh-gradient (`columns`/`rows`/`points`/`colors`)
and expo-symbols (`animationSpec` with variable-color sub-specs) show the shipped bar.
What fx exposes today is `effect: string` + one shared `intensity`, with
`materialConfig`/`symbolConfig` as hand-rolled exceptions (`runtime/FxHostedView.tsx`).
The `02` open question ("where does per-effect typed config live") is unresolved while
being the most user-visible question in V1; palettes/themes attach at the same point and
are blocked behind the same gap.

**Alternative:** resolve the way `02` already leans — canonical uniforms in the manifest
data (F3), TS types generated from it; the same generated table drives the F3 conformance
test. One mechanism, two findings closed.

### F8 — four motion knobs; `tune` is the weak member (MEDIUM)

`preset` (bundle) + `motion` (shape override) + `tune` (intent) + `transition` (timing),
plus `motion` meaning a *states* map on `FxView` but a *phases* map on `FxPresence`.
Comparison mental models: Reanimated 4 ships two concepts (entering/exiting/layout +
CSS-style transitions); Framer ships variants + transition; Rive ships inputs. The
`preset`/`motion`/`transition` triad is load-bearing — the law needs an explicit-override
channel. `tune` is a third adjustment axis whose `{speed, emphasis, distance}` semantics
are invisible until MOT-001's device tuning lands and whose job overlaps both neighbors.

**Alternative:** ship V1 with `preset`/`motion`/`transition`; hold `tune` until MOT-001
proves the formulas. Cutting it now costs nothing — it does not exist yet. Watch-item, not
must-change: the same-prop/different-shape polymorphism of `motion`.

### F9 — `FxGroupView` is a public stub inside the stability contract (MEDIUM)

`src/index.ts` declares its exports "the only stability contract" and ships `FxGroupView`
+ `NativeFxGroupProps`, while the native class is a TODO shell with no child handling and
no glass (`packages/ios/FxGroupView.swift`, 12 lines; Kotlin twin equivalent). A 0.1.x
consumer can mount it and get an inert view.

**Alternative:** hold it out of the public index (or mark `@experimental`) until the morph
compound lands.

### F17/F18 — naming and event seams (LOW)

`<Fx effect={fx.effect.glow()}>` stutters package/component/namespace.
`interactionMode="passive"|"active"|"controlled"` is runtime vocabulary on a user surface.
`hosted`/`expo-view` leak mechanism (and a vendor name) into docs that otherwise refuse to
leak. Native event names (`onShaderPress*`, chosen to dodge RN reserved names) will not
match the public contract (`onPress*`, `onTransitionEnd` per `40`) — the surface
components need the mapping pinned in one place. All cheapest to fix before the surface
freezes.

## Real-world adoption findings

### F13 — coexistence and parity are the adoption gates (MEDIUM)

Every production RN app already ships Reanimated and RNGH. The questions a senior engineer
asks before adopting fx: does `FxPressable` fight RNGH's orchestrator (RNGH's
`waitFor`/`simultaneousHandlers`/`blocksHandlers` relations are the precedent fx's
interactive surface must speak — U8-002)? Can Reanimated animate a view inside an `FxView`
wrapper? What happens to a hosted effect during a native-stack transition? None need
solving now; all need a published compatibility table with honest "untested" cells.
Parity: trace the real outcomes of the ladders — `shader` on Android < 13 →
`{via:'none'}` → a blank hero unless the developer designed fallback content; `symbol` is
iOS-only (DOC-008); glass on Android is blur approximation. DOC-012's degradation policy
is right; `skills/` must lead with a per-capability parity matrix and a "design your
fallback content" pattern. Same bucket: Expo Go cannot load fx (dev build required); web
renders `null` (`runtime/*.web.tsx`) — and the web contract for `FxPresence
visible={false}` (instant unmount vs held forever) must be pinned when the surface
components land.

### F14 — no scale story in the harness (MEDIUM)

Real apps put effects in lists. A `FlatList` of 50 glass cards = 50 hosting controllers
(the known iOS perf trap that motivated `UIHostingConfiguration`); V2 motion on those
cards = 50 MTKViews under F2. The harness (EX-001) has single-instance screens only. Add a
100-cell list stress screen as a standing device-verify scenario — it also converts F1/F2
from theoretical to measured.

### F11 — frame-rate and battery policy is unowned (MEDIUM)

`FxSurfaceView` pins `preferredFramesPerSecond = 60`; hosted iOS runs
`TimelineView(.animation)` at display rate; Android invalidates every Choreographer frame.
The named V1 use cases are ambient backgrounds — continuous full-rate GPU work on screens
users park on, the canonical source of "this library drained my battery" reviews. The
manifest already has a `clock` axis; extend it (or per-effect config) with a cadence hint
(ambient 30 fps; ProMotion opt-in). Small schema change now, breaking later.

### F10 — accessibility beyond reduce-motion (MEDIUM)

DOC-010 covered reduce-motion. Nothing covers: decorative effect views defaulting to
`accessibilityElementsHidden` / `importantForAccessibility="no"` (`FxHostedView` sets
neither on either platform); the V2 wrapper + intermediate container preserving the a11y
tree (RNGH's wrapper carries scars here); what VoiceOver/TalkBack perceives for
interactive glass (`interactive: true`, self-gesturing). Cheapest fix: an a11y row in the
device-verify scenario template so every `device:yes` task answers it.

### F12 — React-semantics checklist for the presence FSM (MEDIUM)

StrictMode double mount/unmount, Fast Refresh remounts, Suspense offscreen-hiding, a
re-render landing mid-exit, FlatList eviction (exit fires on recycle, not removal).
Reanimated has accumulated handling for each (interrupted-exit merging, reparenting
transfer, `skipExiting`). None appear in `35`'s research questions. Add them as explicit
rows before U7-001.

### F15/F16/F19 — ship hygiene (LOW)

`react-native-fx` is unclaimed on npm (404, verified 2026-06-10) — claim it before any
public mention. A simulator smoke lane (Apple-silicon simulators run Metal and SwiftUI;
mount each catalog id, screenshot, diff) would catch the U3-006 blank-on-switch class
without device sessions — device remains the gate for feel/touch/springs. The binding
agent/guide system is also the de-facto human contributor docs; a short human path in the
Contributing Guide keeps drive-by contributions viable.

### Positioning note (no finding id)

The V1 wedge against react-native-skia is the platform-exclusive set — system Liquid
Glass, real materials, SF Symbol effects, native springs by default — not the 10 generic
shaders (commodity; Skia renders equivalents cross-platform with a larger community). Lead
the README/skills story with the exclusives; the shader catalog is supporting cast. The
yearly Apple churn (glass needs iOS 26 + Xcode 26 to compile; springs floor at 17;
MeshGradient at 18) is absorbed structurally by the rung schema — budget the September
treadmill as a standing maintenance cost.

  coordinate-space mismatch solved by descending to the child (`GestureHandler.kt:41-68`).
- API evolution: v1 component-per-recognizer deprecated in source in favor of
  `Gesture.Tap()` builders (`src/handlers/TapGestureHandler.ts:67-68`,

- Feedback precedent for `FxPressable`: platform-specific props (`rippleColor`,

### rive + react-native (+ react-native-runtimes)

- Rive: discrete inputs (`setInputState`/`fireState`), native-owned frame loop, semantic
  events out, native touch (`touchBegan`/`touchEnded` → state machine) — fx's contract,
  shipped (`rive/src/Rive.tsx:373-429`, `rive/ios/RiveReactNativeView.swift:936-984`).
- RN Animated native driver: the original targets-in/native-frames design —
  `CADisplayLink` loop (`RCTNativeAnimatedNodesManager.mm:440-480`), non-layout props
  only; `Animated.event` maps native events to values without JS.
- Pressable feedback: Android ripple is native (`RippleDrawable` + `hotspotUpdate`
  commands); iOS highlight is JS-driven — precedent for fx doing better natively.
- react-native-runtimes: off-thread JS isolation, not a presentation runtime — confirms
  the `05` disposition (naming inspiration only).
