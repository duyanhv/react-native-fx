## Verdict

**Approved.** A single curated `ripple` AGSL sampler (`FxContentDistortion.kt:14-27`,
`uniform shader content;` → `content.eval(distortedCoord)`) is applied to `FxSurfaceView`'s
existing content container via `target.setRenderEffect(RenderEffect.createRuntimeShaderEffect(
runtimeShader, "content"))` (`:169`) — draw-time, so the wrapped RN children are never reparented
and stay tappable. The mechanic is gated `>= TIRAMISU` (`:95`, `:173`); below 33 it is inert and
content renders normally. The surface is the deliberately mechanical `contentDistortion='ripple'`
prop (NOT the long-term public API; high-level sugar deferred). iOS is a clean no-op (the prop is
declared empty). `reviewed` is satisfied by this document.

## The rule-#4 inverse, honored exactly

content-distort is the one effect that runs *over live RN children*, which Android permits and iOS
cannot. The load-bearing claim is that `RenderEffect` is a RenderNode/draw-time concern, so
`dispatchTouchEvent` is unaffected and a child inside the distorted content stays tappable — the
inverse of iOS, where hosting RN content to sample it severs touch (rule #4, `01`). The
implementation never reparents: the effect lands on the container that already holds the routed
children, one draw-time filter on its `RenderNode`. Device-proven in run 1 (Increment/Second
counters both fired *through* the active ripple). iOS stays `out-of-scope` in `02`; the prop is a
no-op there — verified on device (run 1, row 7).

## The device-found defect and its fix

Run 1 (POCO F1 / API 35, build `a89d2fb`) was **6/7 PASS** with one blocking defect: the ripple
never started on first mount or after navigate-back — it only animated after a manual off→on
toggle. Root cause (device logcat + source): `startLoop()` was gated on the *content container's*
attach state, but a `ViewGroup` dispatches its own `onAttachedToWindow` **before** its children
attach, so the `resume()` fired from the parent `FxSurfaceView.onAttachedToWindow` saw the
container still detached and bailed with no retry. Background→foreground worked only because the
view stays attached.

The fix (`495fbe5`, `FxContentDistortion.kt` only, init block `:75-87`) drives the loop off the
**container's own** `View.OnAttachStateChangeListener` — `onViewAttachedToWindow → startLoop()` if
enabled, `onViewDetachedFromWindow → stopLoop()` — the correct attach timing, mirroring how the
sibling `FxSurfaceShaderView` (itself a `View`) self-manages. `ensureShader()` runs in `update()`
(`:104`) before the container attaches, so the shader is ready when the listener fires;
`isLooping`/`stopLoop` keep the listener idempotent with `update`/`resume` (`:137`, `:146`). Inert
below API 33 (`isEnabled` never flips). The fix is exactly +19 lines, one file — no change to the
shader, the touch path, or the pause/resume wiring.
