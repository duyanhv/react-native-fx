## What's right

- Both platforms register `setUniform`/`setHighlight` as `AsyncFunction(view, …)` inside the
  `View(FxSurfaceView.self)` / view block, the exact shape of the shipped, device-proven `snapshot`
  precedent (`FxModule.swift:72`, `FxModule.kt:81`). Rule #7 clean — no SharedObject/JSI.
- `controlled` mode wired through both press handlers: no recognizer, no touch consumption, and
  critically **detach without resetting uniforms** (`FxPressHandler.swift:54` `skipUniformReset`;
  `FxPressHandler.kt:43-49` CONTROLLED skips cancel-side reset path).
- Clobber rule for `intensity` is real and correct on both: iOS `applyResolvedConfig` skips
  `uniforms.intensity` when overridden (`FxSurfaceView.swift:377`); Android skips `setIntensity`
  (`FxSurfaceView.kt:477`) AND `onDraw` prefers `customUniforms["intensity"]`
  (`FxSurfaceShaderView.kt:176`). `null` clears the override on both. Number-only, positional
  `setHighlight`, `[0,1]` y-up UV — all honored.
- TS refactor (types → `FxSurfaceView.types.ts`, re-exported from `FxSurfaceView.tsx` + `index.ts`)
  is behaviour-preserving and back-compat. `FxSurfaceViewRef` is well-documented.

## BLOCKERS — fix-round owed before the device gate

**B1 · `setUniform` is not cross-platform symmetric (rule #6 — iOS/Android are peers).** The two
implementations honour *different uniform sets*, and the executor documented the split rather than
reconciling it:
- Android (`FxSurfaceShaderView.kt:110-122` + the `onDraw` custom-uniform loop `:188-193`) honours
  **any declared uniform** — `setUniform("anyDeclaredName", v)` works.
- iOS (`FxSurfaceView.swift:502-516`) honours **only `intensity` and `pressDepth`**; every other
  name hits `default: break` → silent no-op. The root cause is architectural: iOS writes a
  fixed-layout `FxUniforms` struct via `setFragmentBytes` (`FxSurfaceView.swift:606`) — it
  *cannot* address an arbitrary named uniform without a struct redesign; Android's
  `RuntimeShader.setFloatUniform(name, …)` is dynamic.
- Effect: `setUniform("myGlow", 0.5)` on a BYO shader works on Android, silently no-ops on iOS.
  `structure.{ios,android}.md` even state the two different guards. The spec's "general escape
  hatch over any declared uniform" (README § Scope) is **iOS-infeasible** as written.

**B2 · Android crash on a scalar write to a vector uniform.** `setUniform("touch", 0.5)` passes the
`name in declaredUniforms` guard (`touch` is declared), lands in `customUniforms`, and the `onDraw`
loop calls `currentShader.setFloatUniform("touch", value)` — a single float into a `float2` uniform
→ `IllegalArgumentException` at draw time (`FxSurfaceShaderView.kt:188-193`). The guard checks the
name is declared, never the arity/type. iOS no-ops the same call (safe). A no-op is the contract; a
crash is not.

### Recommended reconciliation (resolves B1 + B2 together)

Bound `setUniform` to an **explicit, known SCALAR set, symmetric on both platforms** — for this cut
`intensity` and `pressDepth` (the only scalars the contract needs; `touch` is set via
`setHighlight`). Drop Android's arbitrary-uniform `customUniforms` loop (or restrict it to the same
known scalar names). Grow the set per feature: **DEF-011 adds its drag/tilt scalars to both the iOS
`FxUniforms` struct + the `setUniform` switch AND the Android known set.** This makes the platforms
peers, kills the crash, and matches what `controlled`/DEF-011 actually needs (named scalars, not
arbitrary BYO uniforms). The spec's "general escape hatch" line narrows to "a known scalar set" —
my edit to make, then a small Android fix-round.

## MINOR — note, not blocking

- **M1 · iOS `pressDepth` override is tracked but never read.** `imperativeOverrides.insert("pressDepth")`
  (`FxSurfaceView.swift:511`) is never consulted — `applyResolvedConfig` checks only `"intensity"`
  (`:377`). Harmless, but `structure.ios.md` overstates "skips reapplying … for any overridden
  uniform"; only `intensity` is a prop-reapplied uniform, so only it needs the guard.
- **M2 · Overrides are never cleared when leaving `controlled`.** Go controlled (override intensity)
  → passive, and the override persists, so prop intensity stays suppressed indefinitely. Likely
  wants a clear-on-exit. Edge case.
- **M3 · `setHighlight` persists by control-flow, not by tracking.** iOS `setHighlight` writes
  `uniforms.touch`/`targetPressDepth` with no `imperativeOverrides` entry (`:525-529`); it survives
  re-render only because nothing in the apply path resets `touch` in controlled mode. The device
  spike must explicitly confirm `setHighlight` survives a Fabric commit (the README spike says
  "setUniform/setHighlight").
- **M4 · `React.ReactNode` unqualified** in `FxSurfaceView.types.ts:` (`children?: React.ReactNode`)
  — the file imports no `React`; compiles via the global, but the original imported `ReactNode`
  directly. Style nit.

## Verdict

**NOT ready for the device gate.** Headless is green, but B1 breaches rule #6 (peers) and the spec
contract, and B2 is a real crash path — by the project's own bar (rules enforced at the `reviewed`
gate) the change is not done. A bounded fix-round (the known-scalar-set reconciliation) clears both.
After the fix: spec narrowed, Android bounded + recompiled, then the device spike (the clobber rule)
is the human's gate.
