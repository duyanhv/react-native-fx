# U8-002 · the RNGH coexistence matrix + cancel path — review

Date: 2026-06-13
Branch: `integration/0.1.x`. Scope: the example coexistence harness + the device matrix vs the
task contract (`tasks/U8-002/README.md`); evidence at `evidence/device.md` (+ `headless.md`,
`android-crash-tombstone.txt`, the iOS/Android screenshots). The planner thread reviewed and
wrote this; the device runs + the close-with-waivers decision were the maintainer's. Closes
**RT-001**.

## Verdict

**Approved — RT-001 closed with documented waivers.** The harness (example-only; the library
stays RNGH-free, rule #7) drops one `active`/`passive`/`controlled` `FxSurfaceView` into every
gesture container the RN ecosystem ships — native ScrollView, pager-view, `@gorhom/bottom-sheet`,
a raw RNGH `GestureDetector(Pan)`, and nested — over a timestamped semantic-press log. Across the
two platforms every distinct mechanism RT-001 names is device-proven; the remaining rows are
per-platform-redundant and waived explicitly. `reviewed` is satisfied by this document.

## What the matrix proved (combined platforms)

- **tap → one `onPress`** — iOS 1a/2a; Android 1a/2a/3b/4-tap.
- **scroll cancel** (PressIn/PressOut, no Press; scroller moves) — iOS 1b; Android 1b (inner)
  + 1d (outer).
- **pager cancel** — iOS 2b.
- **`@gorhom/bottom-sheet` RNGH-pan cancel — the "hard case" `30` flagged** — Android 3a. The
  recognizer is cancelled cleanly by an RNGH pan, the single most load-bearing coexistence claim.
- **`passive`/`controlled` silence** — iOS 5/6; Android 5/6 (handlers wired, so an empty log is a
  positive result).
- **nesting** — root-caused (below).

## Waivers (per-platform-redundant, documented)

Each waived row is the same recognizer mechanism already proven on the other platform, not a new
risk:

- Android **1c** (hold-still tap), **2b** (pager), **4** (raw RNGH `Pan`) — needs-manual under
  agent-device because the cancel rows need a faithful "press-then-drag-past-slop" the tooling
  could not reproduce; the pager mechanism is proven on iOS 2b and the RNGH-pan mechanism on
  Android 3a (`@gorhom` is an RNGH pan).
- iOS **1c/1d/3/4/7** — the curated stitchable Metal shaders render only on a physical iPhone, so
  the simulator could not support visual targeting of the sheet/RNGH/nested surfaces; the iOS
  press events fired correctly on the invisible-but-present views, and the mechanisms are proven
  on Android.

The honesty discipline held throughout — no row the runner could not faithfully exercise was
scored PASS.

## The nesting finding (row 7) — root-caused, policy ratified

With the inner surface re-pointed to `dots` (distinct from the outer `aurora`) and re-observed on
API 33: a shader-bearing surface nested inside another is **occluded and touch-shadowed by the
outer** — `FxSurfaceView` composites its shader view above its content container, so the outer's
shader paints over the inner (which renders the outer's bands, no `dots` grid) and the outer's
recognizer claims the touch (`nested·OUTER` logs; `nested·INNER` never does). This confirms `30`
Decision 6 (the surface is ONE interactive unit); the policy is ratified there — the outer is the
single interactive unit, shader-in-shader is not a V1 composition pattern, use layered
composition.

## The crash detour (U8-001 → U8-003)

The first Android run hard-blocked: the coexistence screen was the first to mount `active` shader
surfaces at scale and tripped a latent API-33 native crash in U8-001's interactive-uniform probe.
The freeze held (the runner flagged it, did not patch `packages/`); it was routed to **U8-003**
(fixed + device-proven, repro-validated on an API-33 AVD — `reviews/U8-003.md`), after which the
Android matrix ran. The harness doing exactly this — surfacing a latent defect the unit's own
API-35 gate missed — is the value of the coexistence matrix.
