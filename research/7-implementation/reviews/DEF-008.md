## The spike (step 1) — resolved NO, headless-definitive

SwiftUI `.colorEffect` cannot consume a runtime-built `MTLLibrary`: `ShaderLibrary`'s only public
initializers are `.default` / `init(url:)` / `init(data:)` over a *compiled* `.metallib`, with no
init over `MTLLibrary`/`MTLFunction`/MSL source (iOS 26.5 SDK), and `makeLibrary(source:)` yields a
library with no path back to metallib `Data`. An API-surface fact, not a render quirk. Consequence,
correctly implemented: runtime shaders lower through the **expo-view** Metal path
(`MTLDevice.makeLibrary(source:)` → pipeline) even for decorative use. Recorded in
`structure.ios.md` § shader.

## Review refinement (maintainer-flagged, applied 2026-06-14)

1. **Naming.** `resolvedPipeline(for:)` hid a runtime-compile side effect behind a lookup-sounding
   name. Renamed `pipeline → curatedPipeline`, `resolvedPipeline → pipeline`; the resolver's doc now
   flags the side-effect compile.
2. **Curated-id collision, native-enforced.** The runtime fallback after a nil curated lookup could,
   in principle, let a registry source satisfy a curated id with no raster fragment — unless the
   registry never holds a curated id (which the JS guard ensures). Hardened: `pipeline(for:)` now
   stops on curated ids via an iOS `curatedShaderIds` mirror (matching Android's `CURATED_SHADER_IDS`
   set), so "curated ids win" is native-enforced, not JS-dependent. **Behavior-preserving** — a
   curated id with a registered source is impossible via the JS guard, so no tested output changes;
   iOS xcodebuild re-confirmed.
