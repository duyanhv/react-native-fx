drives fx's own hosted SwiftUI tiles via `.scrollTransition`, with zero per-frame JS *and* zero

mapping off the JS thread) are proven, the surface is rule-clean, and the manifest models

## Two spec deviations — both ratified

1. **Container-scoped `source` with tiles-as-data, not a per-`<Fx>`-item prop.** The spec said
   "source binding on `<Fx>`"; the executor's choice is *more* rule-correct — per-item `<Fx>`
   children would mount RN content inside a hosted SwiftUI `ScrollView`, the exact rule-#4
   severing the mechanic pin forbids. Tiles-as-data is forced by rule #4. The spec phrasing was
   looser than the rule permits; the executor corrected toward the rule. Ratified into `50`
   Decision 9.
2. **`Fx` as a namespace object (`{ Scroll }`).** Provisional, forward-compatible with the
   DEF-015-frozen `<Fx effect>` compound (becomes a callable component carrying the same
   `.Scroll` static when the effect surface lands). No conflict today. Ratified into `50`
   Decision 9.

## Rules gate (independent)

- **#1** only `axis`+`tiles` cross the bridge, once per change via `applyResolvedConfig`;
  `.scrollTransition` runs in the render server; no per-frame dispatch. Row 2's *structural*
  proof is decisive — the surface exposes no scroll callback, so nothing *can* fire per frame.
- **#2** agnostic names (`source`/`scroll`); the default mapping is SwiftUI's own edge
  transition (the law).
- **#3** `interaction:'self'`; the hosted `ScrollView` self-gestures.
- **#4** tiles are fx-owned data dispatched to `FxFillView`/`FxShaderView`/`FxEmptyView`, never
  RN children — the load-bearing correctness point of the whole design.
- **#7** pure Swift `UIHostingController` + Expo Module DSL; no JSI/C++.
- **#9** reads scroll geometry, writes no layout.
