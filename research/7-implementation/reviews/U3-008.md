## Done-when reconciliation (all met)

- **One persistent controller** — `FxHostedView.hostingController` is created once in
  `mountHostIfNeeded()` (`FxHostedView.swift:101`) hosting `FxHostedRootView` over the
  `FxHostedProps` observable holder; the controller type narrowed from
  `UIHostingController<AnyView>` to `UIHostingController<FxHostedRootView>`.
- **Batches mutate, never remount** — `applyResolvedConfig()` writes the four holder fields after
  `mountHostIfNeeded()`; the per-batch `mountHost(AnyView)` teardown is gone. Mutation is once per
  Expo prop batch on the main actor — never per frame (rule #1).
- **Mount paths stay exclusive** — `mountHostIfNeeded()` opens with `removeGlassSurface()`;
  `mountGlassSurface()` opens with `removeHost()` (`FxHostedView.swift:102,126`). Crossing
  realization class is the only inter-effect controller teardown; the glass surface is itself
  reused in place when already mounted, so glass press state survives config changes.
- **Symbol precedence preserved** — a symbol config rides the SwiftUI path even when `effect` is
  also set; only a symbol-free `material` crosses to UIKit glass (`FxHostedView.swift:67`,
  mirrored in `FxHostedRootView.body`).
- **A11y defaults (F10's cheap half)** — decorative hosts exit the accessibility tree on both
  platforms: `controller.view.accessibilityElementsHidden = true` (`FxHostedView.swift:110`),
  `FxGlassSurfaceView` hidden at init and re-admitted via `accessibilityElementsHidden =
  !newInteractive` (`FxGlassSurfaceView.swift:33,58`), Android
  `importantForAccessibility = IMPORTANT_FOR_ACCESSIBILITY_NO` (`FxHostedView.kt:105`).
- **Mechanic pinned** — `structure.ios.md` §Hosting mechanics gains the persistent-host and
  accessibility-default bullets, including the open interactive-glass VoiceOver item routed to
  the interactive-surface work.
- **Guide row** — the Device Verification Guide's expectations gain the VoiceOver/TalkBack row
  ("every `device: yes` scenario answers this").
- **Reference fidelity** — adopts exactly the SwiftUIHostingView idiom named ADOPT (persistent
  controller + observed props holder); the safe-area machinery and pre-16.4 subclass hack named
  REJECT are correctly absent.

## Nits (non-blocking)

1. **`FxHostedProps.materialConfig` is currently inert** — published but never read:
   `FxHostedRootView`'s below-26 `material` case constructs `FxMaterialView(intensity:)` only,
   so variant config is ignored on the SwiftUI fallback path. Pre-existing behavior (the old
   `makeSwiftUIView` did the same); the field is forward-provisioning. Fine to keep; flagged so
   the eased-`transition` work doesn't assume it's wired.
2. **Teardown wording nuance** — `structure.ios.md` says the controller is torn down "only when
   the effect identity changes its class of realization … or the host unmounts"; the
   both-props-nil branch (`FxHostedView.swift:69-72`) also tears down. Defensible reading
   (effect → none is a realization-class change); no edit requested.
