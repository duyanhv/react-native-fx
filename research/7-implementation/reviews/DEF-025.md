## What's right (spec-faithful, confirmed by code-read)

- `registerSymbol` mirrors `registerShader`: JSON crosses the bridge once keyed by name, per-view
  only `name` crosses; iOS registration is a no-op; guards (empty/missing payload, idempotent
  re-register, replace-on-changed-source); `isRegisteredSymbol`/`registeredSymbolNames`; exported
  from `index.ts`. `type:'lottie'` discriminant only.
- Builder trigger-normalization confined to `stack.ts` (both `.symbol()` and standalone `symbol()`),
  fills `trigger` only when omitted (discrete→`'value'`, indefinite→`'repeat'`); manifest + native
  literal defaults untouched. Also fixes the pre-existing iOS gap where indefinite animations needed
  an explicit trigger.
- Feature detection is narrow (not coordinator-shaped, not DEF-023): `Class.forName` probe →
  module `Constant("features")` → `capabilities.ts` (read once) → `Fx.tsx` passes `features` into
  `select()`. First runtime consumer of the `requires.feature` mechanism (SPINE-005).
- Scope guards hold: no AVD, no bundled icon pack, no `androidAsset` prop, no Compose, detection
  only for `lottie`, iOS symbol behavior unchanged beyond the shared builder default.

## Findings and dispositions

1. **0×0 render blocker** (`FxSymbolView` FrameLayout-with-child never measured its Lottie child).
   First fixed by subclassing `LottieAnimationView` — see finding 2 for why that was reverted.
2. **S4 class-load crash (device-found blocker).** Clean device test: with the Lottie peer absent
   the app crashed `NoClassDefFoundError: ...LottieAnimationView` at Fabric `preallocateView` →
   `FxSymbolView.<init>`, before `onError`. Root cause: subclassing `LottieAnimationView` made merely
   constructing/loading `FxSymbolView` link the absent superclass; Fabric preallocates before the
   selector, and the example's "Direct" column drives `FxHostedView` (a public export) past the
   `<Fx>` selector — so the selector can never be the sole guard. **Resolution:** `FxSymbolView`
   reverts to a `FrameLayout` (always constructible), creates the Lottie child lazily behind a
   one-time `Class.forName` (`lottieAvailable`) guard, and fixes 0×0 via an `onLayout` child
   measure+layout. Re-gate: S1–S4 + off-window all PASS. The planner's earlier "recommend against
   native defense-in-depth" was wrong — the maintainer's call for a native guard was correct.
3. **Stale-composition cache key** → `fromJsonString(json, null)` (re-parse on re-registration).
4. **Per-batch replay/re-warn** → `AppliedConfig` idempotence guard in `configure`.
5. **Loop keyed on trigger alone** → keyed on animation class (`isIndefinite`), iOS-parity.
6. **`Constants` DSL deprecated (iOS + Android)** → `Constant("features") { … }`; surfaced only by
   the native compiles, not lint/swift:lint.
7. **Lottie probe caught only `ClassNotFoundException`** → also catches `LinkageError`, so a
   half-present peer can never take down module load.

Findings 3–7 were applied by the planner/subagent during review and re-verified by code-read + gates.
