## Blockers — the unit cannot run

- **B1 — `FxPressableView` registered in neither module.** `FxModule.swift` (Hosted/Surface/Scroll/Group) and `FxModule.kt` (Hosted/Surface/Group) do not declare `FxPressableView`; JS calls `requireNativeView('ReactNativeFx', 'FxPressableView')` → runtime lookup fails on both platforms.
- **B2 — the recognizer is never activated.** No `pressHandler.update(mode:"active")` exists for `FxPressableView` on either platform (zero matches). iOS never `attach()`es the recognizer; Android `handle()` early-returns while `mode == NONE`. No events, no feedback — content press is always-on and must put the handler into active mode.
- **B3 — iOS effect-surface regression (passive fires `onPressIn`).** `FxSurfaceView.handlePressBegin` dispatches `dispatchShaderPressIn` unconditionally, before the `guard mode == .active`; the original dispatched it only when active. Violates `30` (passive = no `onPress*`). **Android guards this correctly** (`if currentInteractionMode == "active"`) — iOS must match.
- **B4 — Android effect-surface regression (drag-tilt never settles).** The original `handleUp`/`cancel` called `settleDragTilt()` to reset the tilt uniforms on release/cancel; the new `FxSurfaceView.handlePressEnd`/`handlePressCancel` omit it. `dragAxis`/tilt (DEF-011) won't return to neutral after a drag. **iOS preserved this** — Android must restore it.

## Quality / verify (same round)

- **Q1 (iOS feedback)** — `CASpringTimingFunction` on `CAAnimationGroup.timingFunction` is not a real per-property spring; the `asyncAfter(0.4s)` transform reset clobbers a rapid re-press within 400 ms. Prefer `CASpringAnimation` per property with clean cancellation/state tracking.
- **Q2** — Android `handlePressBegin` adds `updateEffectSurfaceVisibility()`, absent from the original `handleDown`; confirm intent (not a surface behavior change).
- **Q3** — passive uniform-not-reset-on-lift (both platforms; the early-return skips the original reset). Minor.
- **Q4** — Android `dispatchTouchEvent` returning `true` blocks the wrapped child's own touches once activated; acceptable for presentational content in V1, but make it deliberate.

## Credit / accepted as-is

- Internal-id discipline clean in shipped code.
- The host-protocol shape (`FxPressHost`: hit-target / begin-change-end-cancel / long-press) is sound; the iOS FSM refactor is otherwise faithful (one passive bug).
- Structure-doc mechanics pinned with real content (host protocol documented both platforms).
- Harness screen (`example/screens/fx-pressable.tsx`) wired into the router correctly.

## Disposition

- Back to the executor for a fix round (substantial native work across both platforms — beyond a bounded review-stage fix). Re-spec unchanged; the spec was correct, the implementation diverged.
- Hygiene handled by the planner this review: untracked `docs/superpowers/plans/task-2-report.md` + gitignored `.superpowers/` and the per-task reports; corrected `example/data/tasks.ts` (U13 `verified`→`in-progress`; U11 `in-progress`→`verified`).

# Review — U13-001 (round 2 + planner fix)

Reviewer: planner · Date: 2026-06-20

The executor's fix work continued past the commit reviewed above (HEAD reached `671922f`). On re-review of the live tree, the earlier blockers were resolved (recognizer activation, the iOS `EventDispatcher` `.dispatch` bug, the passive `onPressOut` guard, the Android drag-tilt reset), but three issues remained — two raised by the maintainer's own sweep:

- **High — the event pipeline was dead.** A commit (`1e79bef`) removed the `Events(...)` declarations from both module `View` blocks to dodge React Native's reserved `topPress` conflict. That was the wrong fix: Expo wires `EventDispatcher` fields only against `viewDefinition.eventNames` (iOS) / `callbacksDefinition` (Android), so with no `Events(...)` the view mounts but every `onPress*` callback is a no-op.
- **Medium — Android ripple radius** computed in `init` (pre-layout, 0-size → locked to 1px).
- **Low — iOS press-down** started from `1.0` rather than the current presentation value, so a re-press during spring-back could jump.

Also found: across rounds 1–2 the iOS `FxPressableView.swift` had **never actually compiled** — it wasn't in the iOS build's source set (no `pod install` after the file was added), so the executor's "iOS BUILD SUCCEEDED" never exercised it. After `pod install`, it surfaced the `.dispatch` errors (since fixed) and then compiled.

**Planner fix applied** (`7e3e036`, maintainer-directed — bounded, fully specified): native events renamed to `onFx`-prefixed and re-declared via `Events(...)` in both modules; public `onPress*` props map onto them in `FxPressable.tsx` (mirroring the shader surface's `onShaderPress*`); Android ripple radius moved to `onSizeChanged`; iOS press-down reads current presentation values.

**Gates re-run (independent):** packages lint/build/138 tests/example tsc; iOS `xcodebuild` BUILD SUCCEEDED (with `FxPressableView.swift` genuinely compiled); Android `compileDebugKotlin` + `assembleDebug` BUILD SUCCESSFUL.

**Disposition:** the unit is now headless + build green on both platforms with the event pipeline structurally correct. **The device feel-gate is still owed** — it must confirm the callbacks actually fire (the event-pipeline fix only proves at runtime), plus feedback feel, 4-event order, long-press suppression, scroll-yield (`onPressOut` only), and `<Fx interactionMode>` no-regression. Nothing pushed.
