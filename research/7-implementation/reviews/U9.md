# U9 · the runtime object model — V1 ratification

Date: 2026-06-13 · Type: `ratify`/`doc-cleanup` (no code, no device gate)
Branch: `integration/0.1.x`. Planner ratification; maintainer chose this path ("close Unit 9
honestly for V1") after the scope audit. Closes **RT-008** and **SPINE-009**.

## Why this is a ratification, not a build

Unit 9 (`doc 36`, blueprint) is marked Phase v2. The audit (2026-06-13) found that U9-001's
headline deliverable — wire `Fx*` `SharedObject`s — **has no V1 consumer and no object that
needs it**:

- The runtime objects ship as plain `internal`/`final` native classes (`FxAnimationDriver` U6,
  `FxPresenceCoordinator` U7, `FxLayoutObserver` U5, `FxPressHandler` U8). They never cross the
  JS boundary, so they are correctly *not* `SharedObject`s and not yet HybridObject-shaped.
- `FxEffectRenderer` was never built as a discrete object; effect rendering lives inline in the
  view classes.
- V1 exposes zero JS-held handles — the only imperative call is `snapshot`, which returns a
  *value*. A `SharedObject` is a JS-held handle; with no holder, there is nothing to wire.

Building the SharedObject layer now would be speculative architecture ahead of any V1 need —
against the "close out V1" objective. So Unit 9's V1 outcome is to **ratify the object graph
as built** and **device-settle the architectural bet it exists to protect**, deferring the
SharedObject construction to when an imperative surface needs it.

## RT-008 — runtime-object granularity (closed)

Finalized in `36` §Resolved questions against the built code: **one** `FxAnimationDriver` with
two families (content shipped, effect V2), never two objects; **per-view native clocks**
(`CADisplayLink`/`Choreographer`, paused off-window), no global scheduler; the discrete
`FxEffectRenderer` object + the decorative-vs-interactive split deferred to **DEF-020**. These
were the only open design questions in `36`; the lean ("one driver, two families, shared
scheduling") matched what U6 built, so this is recording reality, not deciding anew.

## SPINE-009 — the no-Nitro falsification test (closed, device-proven by citation)

SPINE-009's close condition: *on-device proof that coordinator/wrapper identity holds across
Fabric commits, and that fx reads the post-layout frame natively.* Both halves are met on
hardware across the maintainer-ratified U5–U8 gates — this is the REAL-001 move (close by
citation to prior device runs), and it is honest because the condition is genuinely satisfied,
not merely provisioned:

- **Native layout read:** U5-001/RT-013 device-proved the post-layout frame read on
  `FxSurfaceView`, Fabric-only.
- **Identity across Fabric commits:** the driver held identity across retargets
  (U6-001/U6-002); the coordinator held identity + state across re-render-mid-exit, StrictMode
  double-invoke, and list eviction (U7-001/U7-002); the recognizer + the surface held identity
  across the touch path, including the `none`-mode hit-test reshaping (U8-001).
- **The constraint** (Fabric re-applies `transform`/`opacity` → animate an fx-owned
  intermediate layer) was built in U4 and device-proved.

**Coverage check (no gap):** every runtime object that exists has had its identity exercised on
device. The only scenario not covered is a JS-*held* reference called across many commits — that
is the SharedObject/imperative case, which is V2 (DEF-020). For the V1 object model (no JS-held
handles), SPINE-009 is fully proven. `SPINE-010` is not triggered; `05`'s falsification open
question flips from source-audit PASS to device-proven. **The whole no-Nitro bet is
device-settled for V1.**

## Deferred (DEF-020)

The `Fx*` `SharedObject` layer + the discrete `FxEffectRenderer` object + the HybridObject
*shape* (`equals`/`dispose`/identity). Trigger: the first imperative JS-held handle
(controlled-mode `setUniform`/`setHighlight`, a runtime controller) or per-child control (the
`05` Nitro-reconsideration trigger). The HybridObject shape keeps a later Nitro swap mechanical.
